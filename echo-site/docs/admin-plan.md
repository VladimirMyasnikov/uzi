# План админ-панели для витринного сайта УЗИ

## Текущее состояние

- **Каталог:** данные в `catalog-data/` — папки категорий, внутри папки товаров с `data.json` и изображениями (jpg/png/webp). Загрузка через `catalog-loader.ts`, отдача картинок — Route Handler `/images/[...path]`.
- **Контент:** спецпредложения, новости, FAQ, услуги — в `src/config/data.ts` (статический TS).
- **Формы:** запрос КП и контакты — статичный HTML, бэкенда нет.

---

## Цели админки

1. Управление каталогом: категории, товары, описания, изображения.
2. (Опционально) Редактирование контента: спецпредложения, новости, FAQ.
3. (Опционально) Просмотр лидов с форм после появления бэкенда для форм.

---

## 1. Архитектура

### Вариант A: Админка внутри того же Next.js-приложения (рекомендуется)

- **Плюсы:** один деплой, общий код (типы, catalog-loader), быстрый старт.
- **Минусы:** нужна защита маршрутов и разделение бандла (ленивая загрузка админ-страниц).

**Структура:**
- `src/app/(admin)/admin/` — route group, все под `/admin/*`.
- Middleware или проверка в layout: доступ только для авторизованных.
- API Routes в `src/app/api/` для операций с каталогом (чтение/запись fs) и, при необходимости, для лидов.

### Вариант B: Отдельное приложение (например, отдельный репозиторий/поддиректория)

- Имеет смысл, если админку будут развивать отдельно или деплоить на другой домен с другими правами доступа к fs/БД.

**Для текущего проекта достаточно варианта A.**

---

## 2. Авторизация

- **Минимум для старта:** один логин/пароль или секретный ключ в env.
  - Реализация: NextAuth.js (Credentials) или простая проверка по cookie/сессии после проверки пароля на API (хеш в env или в одном JSON-файле).
- **Позже:** несколько ролей (редактор каталога / только просмотр лидов), OAuth или провайдеры по необходимости.

**Рекомендация:** NextAuth.js с Credentials Provider, пользователь/хеш пароля в `.env` (или один JSON-файл вне репозитория). Сессия в cookie, защита всех `/admin` и `/api/admin/*` через middleware или проверку в layout.

---

## 3. Функциональные блоки

### 3.1. Каталог (приоритет 1)

Данные живут в fs: `catalog-data/<categorySlug>/<productSlug>/data.json` + файлы изображений.

- **Категории**
  - Список категорий (чтение через существующий `getCatalogCategories` или аналог по fs).
  - Создание категории = создание папки в `catalog-data` с корректным slug (латиница/цифры/дефис).
  - Переименование категории = переименование папки (осторожно с путями в `data.json` не трогать, slug категории берётся из имени папки).
  - Удаление категории = удаление папки (с подтверждением).

- **Товары**
  - Список товаров по выбранной категории (из каталога или прямой обход папки).
  - Добавление товара: создание папки `<productSlug>`, создание `data.json` с полями `category`, `product`, `desc`, при необходимости `url`, `images`, `badge`, `status`.
  - Редактирование: изменение `data.json` (название, описание, badge, status, порядок/список изображений в `images`).
  - Удаление товара: удаление папки товара (с подтверждением).

- **Изображения**
  - Загрузка файлов в папку товара (multipart в API Route, сохранение в `catalog-data/<category>/<product>/`).
  - Список изображений: локальные файлы (как в catalog-loader) + опционально ссылки из `data.images`; отображать превью и порядок.
  - Удаление: удаление файла с диска и при необходимости удаление ссылки из `data.images`.
  - Переупорядочивание: менять порядок в `data.images` или договориться, что порядок = сортировка по имени файла (как сейчас в loader).

Ограничения по fs: все операции только в пределах `catalog-data` (проверка пути через `resolveCatalogRoot` и нормализацию, без выхода выше корня каталога).

### 3.2. Контент: спецпредложения, новости, FAQ (приоритет 2)

Сейчас контент в `src/config/data.ts`. Варианты:

- **A)** Перенести в JSON (например `content-data/specials.json`, `news.json`, `faq.json`) и загружать их в приложении вместо констант из `data.ts`. Админка редактирует эти JSON через API (чтение/запись fs). Плюс: единый подход с каталогом, без компиляции. Минус: рефакторинг страниц на чтение из fs/конфига.
- **B)** Оставить в коде; в админке только просмотр/копирование текстов, редактирование — вручную в репозитории. Минимум реализации.

**Рекомендация:** по мере необходимости ввести JSON в `content-data/` и один раздел админки «Контент» с формами для спецпредложений, новостей и FAQ.

### 3.3. Лиды (приоритет 3)

Сейчас формы не отправляют данные никуда. Чтобы видеть лиды в админке:

- Добавить API Route для отправки формы (например `POST /api/leads` или `/api/quote-request`): валидация, сохранение в хранилище.
- Хранилище на старте: JSON-файл (например `data/leads.json`) или SQLite (better-sqlite3 / D1 на Vercel). Для витрины с небольшим потоком лидов JSON достаточно.
- В админке: страница «Лиды» — таблица (дата, имя, контакты, сообщение, источник страницы), фильтр по дате, экспорт CSV при желании.

Реализацию форм на сайте (fetch на новый API) и страницу лидов можно запланировать сразу после базового CRUD каталога.

## 4. Технический стек админки

- **Роутинг:** Next.js App Router, route group `(admin)`.
- **UI:** можно оставить Tailwind; для форм и таблиц — свои компоненты или лёгкая библиотека (например shadcn/ui или Radix) для модалок, таблиц, тостов.
- **API:** Route Handlers в `src/app/api/`:
  - `GET/POST /api/admin/catalog/categories` — список, создание.
  - `GET/PUT/DELETE /api/admin/catalog/categories/[slug]` — переименование, удаление.
  - `GET/POST /api/admin/catalog/categories/[slug]/products` — список товаров, создание товара.
  - `GET/PUT/DELETE /api/admin/catalog/categories/[category]/products/[product]` — редактирование/удаление товара.
  - `POST /api/admin/catalog/categories/[category]/products/[product]/images` — загрузка изображения.
  - `DELETE /api/admin/catalog/.../images/[filename]` — удаление изображения.
- Все маршруты под `api/admin` проверять на наличие авторизованной сессии (и при необходимости роль).

---

## 5. Безопасность

- Все пути при работе с fs строить от `resolveCatalogRoot()` и проверять `path.normalize(...).startsWith(catalogRoot)`. Не принимать произвольные пути из запроса.
- Загрузка файлов: проверять MIME/расширение (только jpg, jpeg, png, webp), ограничить размер (например 5–10 MB).
- Админские API и страницы `/admin` недоступны без авторизации (middleware или layout).
- В продакшене HTTPS и секреты только в env.

## 6. Поэтапный план внедрения

| Этап | Задачи |
|------|--------|
| **1. Основа** | NextAuth (Credentials), страница входа, layout `/admin` с проверкой сессии, middleware или guard. |
| **2. Каталог: чтение** | API и страницы админки: список категорий, список товаров категории, просмотр одного товара (поля + список изображений). |
| **3. Каталог: категории** | Создание/переименование/удаление папок категорий (API + UI). |
| **4. Каталог: товары** | Создание товара (папка + data.json), редактирование data.json, удаление товара. |
| **5. Каталог: изображения** | Загрузка, удаление, при необходимости порядок (data.images). |
| **6. Контент (опционально)** | Вынести спец/новости/FAQ в JSON, API + формы в админке. |
| **7. Лиды (опционально)** | API приёма форм, сохранение в JSON/SQLite, страница «Лиды» в админке. |

После этапов 1–5 админка уже позволяет полноценно вести каталог без правки файлов вручную. Этапы 6–7 добавляются по необходимости.

## Реализовано (актуально)

- Этапы 1–6 реализованы. Вход: `echo-site/.env.local` с `AUTH_SECRET` и `ADMIN_PASSWORD`, затем `npm run dev`, страница `/admin/login`. Логин — любой, пароль — из env.
- На маршрутах `/admin/*` не показываются хедер и футер основного сайта (компонент `LayoutContent`).
- **Slug** в UI пояснён: часть URL товара/категории (латиница, цифры, дефис); после создания не меняется. Добавление товара: форма «Добавить новый товар» (slug + название); редактирование — по ссылке «Редактировать» в таблице товаров.
- Параметры маршрутов с кириллицей в API и на клиенте обрабатываются через `decodeURIComponent` для корректного отображения и запросов.
